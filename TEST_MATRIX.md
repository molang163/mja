# 最小测试矩阵

本文件描述了在发布新版本之前建议执行的一组手动测试场景。它不
包含测试结果，仅作为验证工具行为的指导。建议在 **Manjaro
Linux**（稳定分支）上，使用 **Python ≥ 3.11**，并保证当前用户
可以使用 `podman` 或 `docker` 运行容器。

## 核心场景

| 类别 | 场景描述 |
|---|---|
| **仓库安装/卸载** | 使用 `python -m mja install <包名> --source repo` 安装来自官方仓库的软件包，并通过 `mja remove <包名>` 卸载。验证状态文件中记录正确并且卸载时导出物保留或删除符合预期。 |
| **AUR GUI 安装与导出** | 对至少一个图形界面应用使用 `--source aur` 安装，并使用 `--export auto`。验证自动导出的桌面文件可以正常启动程序；删除该桌面文件后使用 `mja repair export <包名>` 能重新导出；通过 `mja state rebuild` 重建状态后信息仍一致；最后使用 `mja remove <包名> --unexport` 删除应用和导出物。 |
| **AUR CLI 安装与二进制导出** | 为命令行工具（如 `protonvpn-cli`）执行 `--export bin`，检查在指定或自动生成的目录下创建包装脚本；验证脚本运行正常；通过 `mja remove <包名>` 卸载后脚本仍在（默认保留），使用 `--unexport` 选项卸载时应同时删除。 |
| **医生/列表/更新** | 使用 `mja doctor`, `mja list` 和 `mja update` 确认输出格式与状态一致。建议在有网络和断网两种情况下运行 `doctor`，并用 `--fix` 修复损坏的状态文件。 |
| **搜索接口** | 对不存在的包执行 `mja search`，观察未找到时的提示；使用模糊或模棱两可的关键字验证结果排序合理；分别测试 `--repo` 和 `--aur` 选项。 |
| **JSON 输出** | 在以上场景中添加 `--json` 选项，检查返回的结构化数据字段完整且稳定。 |
| **中断处理** | 在安装或更新过程中按下 `Ctrl+C`，确认程序优雅退出并打印 `Interrupted` 而非 traceback。 |
| **离线分发验证** | 在无法访问网络的环境中，验证 `python -m mja` 或通过 `pip install --no-build-isolation .` 安装的命令能够正常启动，并能适当地报告缺乏网络依赖。请注意，搜索、安装 AUR 包和更新等功能在离线状态下将无法完成；重点验证工具的基本 CLI 启动和错误提示，而非所有功能。 |
| **GUI 语言切换** | 启动图形界面后，通过菜单切换语言（中文/English），验证当前页面标题、按钮、表格标题和状态栏文字立即刷新，不需要跳转页面；当前数据和滚动位置应保持。 |
| **安装/卸载脚本** | 在用户主目录中运行 `./install.sh`，检查脚本检测 Python/PySide6 并在 `~/.local/share/mja/venv` 创建虚拟环境；确认 `~/.local/bin/mja-gui` 启动器存在且可以启动 GUI；检查 `~/.local/share/applications/mja-gui.desktop` 导航可添加菜单入口；再次运行安装脚本应覆盖安装而非创建多个副本。随后运行 `./uninstall.sh` 进行普通卸载，确认 venv、启动器和桌面文件被删除但 `~/.local/state/mja` 保留；使用 `./uninstall.sh --purge` 应额外删除状态和缓存目录。 |
| **安装后命令脚本** | 完成安装后，通过 `mja ...` 调用各命令，确保行为与 `python -m mja` 一致。 |

进行这些测试可帮助确认核心功能在典型环境下稳定工作。建议在每次发布前至少覆盖上述场景。
